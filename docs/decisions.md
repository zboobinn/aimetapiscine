### 2026-07-07 — Calculateur expert : logique pure isolée + convention `coverage`
- **Décision** : toute la logique de calcul vit dans `src/features/calculator/` en fonctions pures sans accès réseau ni dépendance UI (`geometry.ts`, `rolls.ts`, `accessories.ts`, `calculate-pack.ts`), orchestrées par `calculatePack()`. Les paramètres ajustables (coefficients de perte, surfaces forfaitaires par escalier, mapping catégorie → grandeur de besoin) sont centralisés dans `calculator-config.ts` ; les coefficients de perte restent surchargeables par env (`LOSS_COEFF_BASE`/`LOSS_COEFF_STAIRS`, 26) mais la lecture de `process.env` se fait UNIQUEMENT côté Server Component (`src/app/calculateur/page.tsx` via `getBusinessConfigEnv()`), jamais dans `features/calculator/`. Convention `coverage` (04) uniformisée : une seule clé par accessoire représentant le rendement d'une unité vendable (`m2_per_unit` pour feutre/colle, `ml_per_unit`/`ml_per_barre` pour PVC liquide/profils/solvant) — l'entrée `COLLE` de `data/catalog.json` a été convertie de `kg_per_m2` (taux d'application) vers `m2_per_unit` (rendement du bidon) pour rester cohérente ; la grandeur de besoin (surface brute vs périmètre) par catégorie est fixée dans `calculator-config.ts`, jamais dans les fonctions de calcul. Formes extensibles via union discriminée `PoolInput` (V1 : rectangle uniquement). Parcours UI en 4 écrans (`src/app/calculateur/calculator-wizard.tsx`, Client Component) : forme, dimensions, escalier, résultat ; état sérialisé dans l'URL (`url-state.ts`, zod) pour un lien partageable qui relance directement à l'étape résultat. Interface d'ajout panier préparée (`cart-draft.ts`, `buildCartDraft()`) sans store/API réels (09 à venir). Tests unitaires Vitest ajoutés (`vitest.config.ts` racine, alias `@` → `src`) : cas nominaux, bornes min/max, arrondis, coefficient avec/sans escalier, aller-retour URL.
- **Motif** : les rendements APF réels ne sont pas encore connus (24, priorité aux tests pour sécuriser les futurs remplacements de coefficients) ; séparer strictement logique pure et UI permet au serveur de rejouer la même logique au checkout (10) sans duplication. La clé `coverage` du schéma (04) est un `record` libre : uniformiser son nommage ne casse pas le schéma Zod et simplifie le calcul générique par catégorie plutôt que du code spécifique par SKU.
- **⚠️ À refaire à réception des fiches techniques APF** : la conversion `COLLE` (`kg_per_m2: 0.3` → `m2_per_unit: 16.6`) est une simple division arithmétique (5 kg / 0,3 kg/m²) qui SUPPOSE un bidon de 5 kg net — ce contenu n'est pas un champ du catalogue (seul `weight_grams`, poids d'expédition, existe) et n'est donc pas garanti exact même pour la donnée de test actuelle. Dès réception des vraies fiches techniques APF, il faudra : (1) confirmer le contenu net réel de chaque conditionnement (colle, PVC liquide, solvant), (2) recalculer `m2_per_unit`/`ml_per_unit`/`ml_per_barre` à partir des taux d'application réels APF, (3) vérifier si tous les accessoires suivent bien un modèle "rendement par unité vendable" ou si certains nécessitent un mapping catégorie→grandeur différent dans `calculator-config.ts` (`ACCESSORY_YIELD_RULES`).
- **Impact** : 08, 24, 26 ; `src/features/calculator/*`, `src/app/calculateur/*`, `data/catalog.json` (COLLE), `vitest.config.ts`, `package.json` (script `test` déjà présent).

### 2026-07-08 — Frais de port : les deux modes coexistent, Corse incluse avec surcoût provisoire (12)
- **Décision** : `getShippingFee(cart, role, shippingAddress?)` (`src/lib/shipping/get-shipping-fee.ts`) reste l'unique point de calcul du port, mais les deux options tarifaires (A « inclus » / B « forfait ») ne sont plus un choix à trancher : les deux coexistent, sélectionnées par `SHIPPING_MODE` (`included` | `flat`, défaut `included`) — donc un seul déploiement peut basculer sans toucher au checkout. Corse tranchée INCLUSE dans la zone de livraison V1 (pas d'exclusion), mais avec un surcoût transporteur ajouté dans les deux modes dès qu'un code postal 2A/2B (`20xxx`) est connu (`SHIPPING_CORSICA_SURCHARGE_CENTS`, montant PROVISOIRE = 3000, à confirmer avec APF). Détection Corse/DOM-TOM centralisée dans `src/lib/shipping/postal-code.ts` (`isCorsicaPostalCode`, `isExcludedOverseasPostalCode` — DOM 971-976 + COM/TOM 98x). `/panier` propose un champ code postal optionnel (`panier-client.tsx`) qui interroge `/api/cart/resolve` (étendu avec `postalCode`/`shipping` dans la réponse) pour une estimation d'affichage (montant, surcoût Corse, message DOM-TOM le cas échéant, délai blind shipping `SHIPPING_DELAY_LABEL`) ; `/api/checkout` revalide indépendamment ce même code postal (rejet 422 si DOM-TOM) et est la SEULE source du montant réellement facturé (`shipping_options` Stripe, `shipping_address_collection: { allowed_countries: ['FR'] }` déjà en place car ces territoires sont des codes pays ISO distincts côté Stripe — GP, MQ, RE, YT…). Limite V1 assumée : le code postal saisi sur `/panier` est une estimation, rien n'empêche techniquement l'acheteur de saisir une adresse différente dans le formulaire Stripe hébergé à l'étape suivante ; le contrôle DOM-TOM ne bloque donc formellement que via `allowed_countries`, le check code postal étant une défense en profondeur côté site.
- **Motif** : le porteur de projet tranche pour la coexistence des deux modes (au lieu d'un choix figé) afin de garder la bascule commerciale ouverte post-lancement ; la Corse est courante en clientèle piscine et son exclusion pure aurait perdu des ventes, d'où l'inclusion avec surcoût plutôt qu'un refus.
- **Impact** : 09, 10, 12, 26 ; `src/lib/shipping/get-shipping-fee.ts`, `src/lib/shipping/postal-code.ts`, `src/lib/env/schemas.ts`, `src/app/api/checkout/route.ts`, `src/app/api/cart/resolve/route.ts`, `src/app/panier/panier-client.tsx`, `.env.example`.
- **Reste à faire** : TOUS les montants restent provisoires tant qu'APF n'a pas transmis ses tarifs réels — `SHIPPING_FLAT_FEE_CENTS` (~40 €), `SHIPPING_CORSICA_SURCHARGE_CENTS` (~30 €), ET le calibrage du mode `included` lui-même (~15 €/rouleau actuellement supposé fondu dans `data/catalog.json`, cf. tarifs de test à remplacer — 04) : si le vrai coût transporteur diffère, le prix produit doit être recalculé, pas seulement l'env `SHIPPING_MODE`. `SHIPPING_MODE` définitif à choisir avant lancement (actuellement `included` par défaut).

- **Décision** : panier client (`src/features/cart/`, store Zustand persisté en `localStorage`) qui ne stocke QUE `{sku, quantity, source: 'catalog'|'pack', packId?}` — jamais de montant. `/panier` (`panier-client.tsx`) envoie ces lignes à la Route Handler `POST /api/cart/resolve`, qui lit le catalogue (`lib/catalog/data.ts`, même source que le reste du site) et renvoie nom, image, disponibilité et prix unitaire déjà résolus. Le rôle tarifaire est déterminé par `resolvePricingRole()` (`src/lib/pricing/resolve-role.ts`), aujourd'hui toujours `"b2c"` (aucun compte pro vérifié ne peut encore exister) : c'est le SEUL point à modifier pour brancher la lecture de `profiles.role` en DB (14) sans retoucher au reste du panier. Ajout au panier depuis le calculateur (`buildCartDraft` → `addPackLines()`, un `packId` par ajout + la query string du calculateur mémorisée pour le lien « Recalculer ») et depuis les fiches produit (`AddToCartButton`, avec cross-sell des accessoires dont la catégorie a un rendement dans `calculator-config.ts` — même logique que 08, suggestion jamais forcée).

Réutiliser le catalogue JSON (comme toutes les pages produit) plutôt que d'interroger Supabase directement depuis cette route évite une divergence de source de vérité tant que 14 n'est pas là.

Idempotence vérifiée par renvoi du même événement (`stripe events resend` / redélivrance CLI) : aucun doublon créé, court-circuit confirmé sur `stripe_session_id` déjà présent. Confirmé que `/api/checkout` ne lit que `{ sku, quantity }` côté client, tout montant étant recalculé serveur.

### 2026-07-08 — Correctif : client Supabase navigateur ne doit jamais passer par `lib/env`/Zod (14, 26)
- **Décision** : `src/lib/supabase/client.ts` (client `createBrowserClient`) lit désormais `process.env.NEXT_PUBLIC_SUPABASE_URL`/`NEXT_PUBLIC_SUPABASE_ANON_KEY` en accès littéraux directs dans ce fichier, avec un throw explicite si absents — il n'appelle plus `getSupabaseEnv()` (`lib/env`, domaine Zod sur `process.env` entier).
- **Motif** : Next.js n'inline les `NEXT_PUBLIC_*` dans le bundle navigateur QUE pour les expressions littérales `process.env.NEXT_PUBLIC_X` repérées statiquement dans le code source à l'endroit même de leur utilisation — jamais pour un objet `process.env` complet passé en argument à une fonction. `getSupabaseEnv()` fait `schema.safeParse(process.env)` : côté serveur ça marche (le vrai `process.env` runtime, avec TOUTES les variables, secrets compris) ; côté navigateur, `process.env` n'est plus qu'un shim ne contenant que ce qui a été littéralement remplacé ailleurs — l'appel échouait donc avec « expected string, received undefined » dès qu'un composant CLIENT (ici `use-auth-user.ts`, ajouté pour l'état connecté/déconnecté du Header) important indirectement `client.ts` était exécuté dans le navigateur. Aucun composant client n'avait auparavant besoin du client Supabase navigateur (le panier resolvait tout via `/api/cart/resolve` côté serveur) — le bug était donc latent depuis 09/10, révélé seulement par 14.
- **Impact** : 14, 26 ; `src/lib/supabase/client.ts`. Règle à retenir pour tout futur code CLIENT ayant besoin d'une variable `NEXT_PUBLIC_*` : accès littéral direct à `process.env.NEXT_PUBLIC_X` dans le fichier client lui-même, jamais via `lib/env` (réservé au code serveur, où `process.env` est fiable).
- **Validé** : `pnpm dev`, page d'accueil chargée sans erreur navigateur ; valeur du projet Supabase retrouvée en clair dans le chunk client compilé (`grep` sur `.next` après requête du chunk), confirmant l'inlining correct.
