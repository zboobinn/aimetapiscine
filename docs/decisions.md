### 2026-07-07 — Calculateur expert : logique pure isolée + convention `coverage`
- **Décision** : toute la logique de calcul vit dans `src/features/calculator/` en fonctions pures sans accès réseau ni dépendance UI (`geometry.ts`, `rolls.ts`, `accessories.ts`, `calculate-pack.ts`), orchestrées par `calculatePack()`. Les paramètres ajustables (coefficients de perte, surfaces forfaitaires par escalier, mapping catégorie → grandeur de besoin) sont centralisés dans `calculator-config.ts` ; les coefficients de perte restent surchargeables par env (`LOSS_COEFF_BASE`/`LOSS_COEFF_STAIRS`, 26) mais la lecture de `process.env` se fait UNIQUEMENT côté Server Component (`src/app/calculateur/page.tsx` via `getBusinessConfigEnv()`), jamais dans `features/calculator/`. Convention `coverage` (04) uniformisée : une seule clé par accessoire représentant le rendement d'une unité vendable (`m2_per_unit` pour feutre/colle, `ml_per_unit`/`ml_per_barre` pour PVC liquide/profils/solvant) — l'entrée `COLLE` de `data/catalog.json` a été convertie de `kg_per_m2` (taux d'application) vers `m2_per_unit` (rendement du bidon) pour rester cohérente ; la grandeur de besoin (surface brute vs périmètre) par catégorie est fixée dans `calculator-config.ts`, jamais dans les fonctions de calcul. Formes extensibles via union discriminée `PoolInput` (V1 : rectangle uniquement). Parcours UI en 4 écrans (`src/app/calculateur/calculator-wizard.tsx`, Client Component) : forme, dimensions, escalier, résultat ; état sérialisé dans l'URL (`url-state.ts`, zod) pour un lien partageable qui relance directement à l'étape résultat. Interface d'ajout panier préparée (`cart-draft.ts`, `buildCartDraft()`) sans store/API réels (09 à venir). Tests unitaires Vitest ajoutés (`vitest.config.ts` racine, alias `@` → `src`) : cas nominaux, bornes min/max, arrondis, coefficient avec/sans escalier, aller-retour URL.
- **Motif** : les rendements APF réels ne sont pas encore connus (24, priorité aux tests pour sécuriser les futurs remplacements de coefficients) ; séparer strictement logique pure et UI permet au serveur de rejouer la même logique au checkout (10) sans duplication. La clé `coverage` du schéma (04) est un `record` libre : uniformiser son nommage ne casse pas le schéma Zod et simplifie le calcul générique par catégorie plutôt que du code spécifique par SKU.
- **⚠️ À refaire à réception des fiches techniques APF** : la conversion `COLLE` (`kg_per_m2: 0.3` → `m2_per_unit: 16.6`) est une simple division arithmétique (5 kg / 0,3 kg/m²) qui SUPPOSE un bidon de 5 kg net — ce contenu n'est pas un champ du catalogue (seul `weight_grams`, poids d'expédition, existe) et n'est donc pas garanti exact même pour la donnée de test actuelle. Dès réception des vraies fiches techniques APF, il faudra : (1) confirmer le contenu net réel de chaque conditionnement (colle, PVC liquide, solvant), (2) recalculer `m2_per_unit`/`ml_per_unit`/`ml_per_barre` à partir des taux d'application réels APF, (3) vérifier si tous les accessoires suivent bien un modèle "rendement par unité vendable" ou si certains nécessitent un mapping catégorie→grandeur différent dans `calculator-config.ts` (`ACCESSORY_YIELD_RULES`).
- **Impact** : 08, 24, 26 ; `src/features/calculator/*`, `src/app/calculateur/*`, `data/catalog.json` (COLLE), `vitest.config.ts`, `package.json` (script `test` déjà présent).

### 2026-07-08 — Frais de port : les deux modes coexistent, Corse incluse avec surcoût provisoire (12)
- **Décision** : `getShippingFee(cart, role, shippingAddress?)` (`src/lib/shipping/get-shipping-fee.ts`) reste l'unique point de calcul du port, mais les deux options tarifaires (A « inclus » / B « forfait ») ne sont plus un choix à trancher : les deux coexistent, sélectionnées par `SHIPPING_MODE` (`included` | `flat`, défaut `included`) — donc un seul déploiement peut basculer sans toucher au checkout. Corse tranchée INCLUSE dans la zone de livraison V1 (pas d'exclusion), mais avec un surcoût transporteur ajouté dans les deux modes dès qu'un code postal 2A/2B (`20xxx`) est connu (`SHIPPING_CORSICA_SURCHARGE_CENTS`, montant PROVISOIRE = 3000, à confirmer avec APF). Détection Corse/DOM-TOM centralisée dans `src/lib/shipping/postal-code.ts` (`isCorsicaPostalCode`, `isExcludedOverseasPostalCode` — DOM 971-976 + COM/TOM 98x). `/panier` propose un champ code postal optionnel (`panier-client.tsx`) qui interroge `/api/cart/resolve` (étendu avec `postalCode`/`shipping` dans la réponse) pour une estimation d'affichage (montant, surcoût Corse, message DOM-TOM le cas échéant, délai blind shipping `SHIPPING_DELAY_LABEL`) ; `/api/checkout` revalide indépendamment ce même code postal (rejet 422 si DOM-TOM) et est la SEULE source du montant réellement facturé (`shipping_options` Stripe, `shipping_address_collection: { allowed_countries: ['FR'] }` déjà en place car ces territoires sont des codes pays ISO distincts côté Stripe — GP, MQ, RE, YT…). Limite V1 assumée : le code postal saisi sur `/panier` est une estimation, rien n'empêche techniquement l'acheteur de saisir une adresse différente dans le formulaire Stripe hébergé à l'étape suivante ; le contrôle DOM-TOM ne bloque donc formellement que via `allowed_countries`, le check code postal étant une défense en profondeur côté site.
- **Motif** : le porteur de projet tranche pour la coexistence des deux modes (au lieu d'un choix figé) afin de garder la bascule commerciale ouverte post-lancement ; la Corse est courante en clientèle piscine et son exclusion pure aurait perdu des ventes, d'où l'inclusion avec surcoût plutôt qu'un refus.
- **Impact** : 09, 10, 12, 26 ; `src/lib/shipping/get-shipping-fee.ts`, `src/lib/shipping/postal-code.ts`, `src/lib/env/schemas.ts`, `src/app/api/checkout/route.ts`, `src/app/api/cart/resolve/route.ts`, `src/app/panier/panier-client.tsx`, `.env.example`.
- **Reste à faire** : TOUS les montants restent provisoires tant qu'APF n'a pas transmis ses tarifs réels — `SHIPPING_FLAT_FEE_CENTS` (~40 €), `SHIPPING_CORSICA_SURCHARGE_CENTS` (~30 €), ET le calibrage du mode `included` lui-même (~15 €/rouleau actuellement supposé fondu dans `data/catalog.json`, cf. tarifs de test à remplacer — 04) : si le vrai coût transporteur diffère, le prix produit doit être recalculé, pas seulement l'env `SHIPPING_MODE`. `SHIPPING_MODE` définitif à choisir avant lancement (actuellement `included` par défaut).

- **Décision** : panier client (`src/features/cart/`, store Zustand persisté en `localStorage`) qui ne stocke QUE `{sku, quantity, source: 'catalog'|'pack', packId?}` — jamais de montant. `/panier` (`panier-client.tsx`) envoie ces lignes à la Route Handler `POST /api/cart/resolve`, qui lit le catalogue (`lib/catalog/data.ts`, même source que le reste du site) et renvoie nom, image, disponibilité et prix unitaire déjà résolus. Le rôle tarifaire est déterminé par `resolvePricingRole()` (`src/lib/pricing/resolve-role.ts`), aujourd'hui toujours `"b2c"` (aucun compte pro vérifié ne peut encore exister) : c'est le SEUL point à modifier pour brancher la lecture de `profiles.role` en DB (14) sans retoucher au reste du panier. Réutiliser le catalogue JSON (comme toutes les pages produit) plutôt que d'interroger Supabase directement depuis cette route évite une divergence de source de vérité tant que 14 n'est pas là.

## 2026-07-10 — Refonte visuelle : cadrage et décisions

**Contexte.** Le site n'a pas de homepage, les pages produit ne donnent pas envie, et le socle fonctionnel (specs 1–17) est en place. Dossier de recherche complet dans `docs/research/refonte-visuelle.md`.

### Décisions prises

**D1 — Direction visuelle : « Nuancier ».**
Éditorial matière. Le nuancier est le seul objet du métier à la fois beau et fonctionnel — c'est ce que le pisciniste pose sur la table de cuisine du client. Le registre « Bureau d'études » (mono, tableaux, filets, schémas cotés) devient un **sous-registre typographique** confiné au calculateur et aux specs, pas une alternative.
*Écartées :* « Bord de bassin » (c'est ce que font Piscinelle et Diffazur, qui sont des constructeurs, pas des marchands — l'émotion outdoor pure produit des visiteurs qui rêvent et un panier vide) et « Bureau d'études » en direction principale (servir la technique comme identité confirme la peur de l'acheteur particulier).

**D2 — Zéro librairie d'animation.**
Ni GSAP, ni Motion, ni Lenis. Après élagage, la liste des animations tient en : reveal, stagger, crossfade, sticky. Les quatre sont natifs. GSAP est gratuit depuis avril 2025 mais reste closed-source, avec une licence interdisant l'usage dans un concurrent de Webflow et résiliable à discrétion — dépendance politique inacceptable sur un projet solo à 5 ans. Motion est MIT mais coûte 30 KB pour du `opacity: 0 → 1`.
*Garde-fou :* un test CI lit `package.json` et échoue si `gsap`, `motion`, `framer-motion`, `lenis` ou `aos` apparaissent.
*Réserve :* si une page de scroll-telling voit le jour un jour, GSAP + ScrollTrigger en `next/dynamic` sur cette route seule, budget 35 KB, jamais dans le layout racine.

**D3 — `animation-timeline` n'est pas Baseline.**
Firefox stable le garde derrière un flag en 2026 (~82 % de support global), contrairement à ce qu'affirment plusieurs articles FR. Les reveals sont écrits sous `@supports (animation-timeline: view())`, avec **l'état révélé comme état par défaut**. On anime depuis l'état final, jamais vers lui. Firefox voit une page statique et parfaitement lisible.

**D4 — Le hero n'a aucune animation.**
C'est le pari du projet. Une image, six pastilles de couleur, deux boutons. Le silence visuel dans un marché qui crie est le positionnement — et c'est accessoirement le meilleur LCP possible.

**D5 — Le calculateur modifie un prix, il ne le révèle pas.**
Le PDP arrive avec un prix calculé sur `DEFAULT_POOL_DIMENSIONS` (8,00 × 4,00 × 1,50 m). Le bouton ATC est actif au chargement. Il ne dit jamais « configurez d'abord ». Un bouton grisé est un mur ; un prix qui se précise est un escalier.
Le moteur de calcul **ne change pas**. Il est déplacé et habillé, pas réécrit.

**D6 — Le prix au m² est obligatoire sur le PDP.**
81 % des sites ne l'affichent pas (Baymard). C'est le seul chiffre qui permet à un pisciniste pro de comparer une offre. Il passe, comme les trois autres valeurs du bloc prix, par la fonction unique de calcul.

**D7 — Trois vecteurs blind-shipping non traités jusqu'ici.**
Métadonnées EXIF/IPTC/XMP des images ; photos et textes d'avis clients ; `brand`/`manufacturer`/`seller` dans le JSON-LD. Aucun n'était couvert par la spec 15 ni la 17. La spec 27 les ferme **avant** tout travail visuel, avec six tests Vitest dont chacun échoue si sa protection est retirée.

**D8 — Numérotation.**
Nouvelles specs 27 → 31. Les specs 18–26 restent à leur place. Le dossier de recherche n'est pas une spec, il vit dans `docs/research/`.

**D9 — Modération a priori des avis avec photo.**
Non négociable. Un client photographie son colis, le carton porte le nom du fournisseur, et le blind-shipping meurt. Aucune automatisation. Si le temps manque sur la spec 31, on coupe les View Transitions, jamais ça.

**D10 — Détection : trois passes, pas une regex.**
Normalisation et frontières de mot sont incompatibles : stripper la ponctuation
détruit les frontières qu'on veut préserver. D'où :
1. collapse des lettres isolées (`a.p.f.` → `apf`, `cap f` inchangé)
2. match sur frontières de mot, séparateurs préservés
3. match substring agressif, **réservé aux tokens ≥ 5 caractères**
Validation au boot : un token < 5 car. est autorisé mais exclu de la passe 3,
avec warning. Chaque passe est une fonction exportée, testée séparément.

**D11 — Limite de couverture acceptée.**
Denylist mono-mot uniquement (un token multi-mots ne peut distinguer
`brumanor scierie` de `brumanorsciences` sans règle fragile). Les mots
génériques du nom fournisseur sont inutilisables : ils censureraient du contenu
légitime. Un avis mentionnant uniquement la partie générique passera le filtre.
Risque résiduel assumé — et raison pour laquelle la modération a priori (D9)
n'est pas négociable.
Cas de test canoniques : `ZVX`, `z.v.x.`, `z v x` matchent ;
`zvxelstrudel`, `capzvx` ne matchent pas.

**D12 — Deux régimes de validation.**
`catalog.json` (notre donnée, commitée, mirrorée) → Zod au chargement, throw,
build cassé. C'est un bug de développeur, il doit être bruyant.
Champs Supabase / avis clients → `sanitizePublicField`, repli + log caviardé.
La page doit vivre.

**D13 — Le seuil de 5 caractères est une règle de distinctivité, pas d'implémentation.**
Un token ≥ 5 car. est assez distinctif pour que toute occurrence, même noyée dans
un mot, soit intentionnelle : `zolvexair` matche. Un token de 3 car. ne l'est pas —
le hasard du français produit des collisions (`apfelstrudel`), d'où l'interdiction
de la passe 3.

**D14 — `SITE_BRAND`, pas un second nom d'entité.**
La spec 27 disait de hard-coder `"aimetapiscine"` dans `brand.name`/`seller.name`.
Erreur : le codebase expose déjà `SITE_BRAND` pour `Organization`/`Website`.
Deux noms d'entité sur le même site est une incohérence que le blind-shipping
n'aurait pas rattrapée. La garde n'est pas la valeur, c'est que `brand.name` ne
prenne **aucun argument**.

**D15 — Conséquence de D13 : `brumanorsciences` matche.**

**D16 — Le critère « grep vide » de la spec 27 est amendé.**
48 occurrences subsistent dans le dépôt, toutes antérieures et toutes couvertes
par l'exception SKU-serveur (23a). Le critère littéral était mal formulé : le
token fournisseur vit légitimement dans des identifiants serveur.
Critère de remplacement, vérifiable : aucune occurrence dans un fichier
atteignable depuis un Client Component.
Preuve : `git grep -ril "<token>" -- "src/**" ":!src/lib/server/**"` → vide.
La vérification sur `.next/static` seule est insuffisante — elle ne couvre ni les
routes à la demande, ni les payloads RSC, ni les messages d'erreur.

**D18 — Risque physique découvert sur le bon de livraison.**
Le BL fournisseur affiche le SKU préfixé `APF-` et un bandeau nommant APF en
clair. Tant qu'il ne va qu'au fournisseur, c'est sans risque. S'il est glissé
dans le colis — ce qui est la fonction usuelle d'un BL — c'est une fuite directe.
Actions : (1) confirmer auprès du fournisseur ce qui entre physiquement dans le
carton ; (2) si nécessaire, scinder en bon de préparation interne / bon de
livraison client à l'entête aimetapiscine ; (3) reformuler le bandeau sans
nommer le fournisseur.
Question ouverte : le BL est-il un « livrable client » au sens de l'exception
23a ? Si oui, il faut un mapping SKU interne → référence publique.
